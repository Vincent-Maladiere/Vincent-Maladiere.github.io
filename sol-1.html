def count_occurences(string : str):
    lower_string = string.lower()
    dict_counter = dict()
    for character in lower_string:
        if character in dict_counter:
            dict_counter[character] += 1
        else:

We see that both values are not stored as exactly equal to 0.1 according to python.

That is because floating-point numbers are represented in computer hardware as base 2 (binary) fractions. In other words, 0.1 gets converted from base 10 to base 2, where it is written as 0.000110011…(0011).

Unfortunately the resulting binary number may not accurately represent the original base 10 number. 0.1 is one example.

A consequence is that, in general, the decimal floating-point numbers we enter are only approximated by the binary floating-point numbers actually stored in the machine.

result_number2 = np.float32(0.1 + 100 - 100)
comparison2 = (result_number2 == np.float32(0.1))
print(comparison2)

# Correct, but not concise:
def numpy_prod_naive(n):
    return 2 * np.prod(np.divide(4*np.square(np.arange(1, n+1)), (4*np.square(np.arange(1, n+1))-1)))

Using numpy is much faster than the for-loop. More generally, one must always try to use numpy rather than for-loops (in the case of numerical computation). Of course, sometimes, you must use for-loops which are a great tool.

For time.time(), the epoch is the point where the time starts, and is platform dependent. For Unix, the epoch is January 1, 1970, 00:00:00 (UTC). The term seconds since the epoch refers to the total number of elapsed seconds since the epoch. Hence, the duration of running some lines of code is the difference between times.

Note that we set a variable n at the beginning of both for-loop and numpy version, so that:

  • we are sure to use the same value of n for both versions
  • it is easy to change the value of n and run computations again

Indeed, the following code:

In [ ]:
2 * np.prod((4 * np.arange(1, 100000)**2) / ((4 * np.arange(1, 100000)**2 - 1)))
Out[ ]:
3.141584799578707

is not distinsguigable from:

In [ ]:
2 * np.prod((4 * np.arange(1, 10000)**2) / ((4 * np.arange(1, 10000)**2 - 1)))
Out[ ]:
3.1415141108281714

while the value of n is different between both code lines.

Surprisingly, np.arange(5).T has the same shape as np.arange(5), because numpy considers row vectors to be the same as column vectors. Transposing 1D-arrays does nothing.

Be careful that the following is not a numpy 1D-array:

In [ ]:

M = array([[0],
       [1],
       [2],
       [3],
       [4]])
np.shape(M) = (5, 1)
np.shape(M.T) = (1, 5)

Note that np.arange(5).T works exactly the same as np.transpose(np.arange(5)) for 2D-arrays, but np.transpose is more general because it allows choosing the axis. However:
